### 22:40 BST — Rewrote `scenarios/` to drive `StepwisePlant.advance()` directly; all 6 scenarios run end-to-end as plant regression tests
- **Author:** Ajay Talati (senior decision: `_plant.py` is essential for MPC, so an end-to-end plant + obs regression check is genuinely valuable; option B from earlier triage = leave them broken was wrong)
- **Assistant:** Claude Code (Opus 4.7, 1M context)
- **Files:** [scenarios/_common.py](scenarios/_common.py) (full rewrite), [scenarios/14d_set_A_healthy.py](scenarios/14d_set_A_healthy.py), [scenarios/14d_set_B_amplitude.py](scenarios/14d_set_B_amplitude.py), [scenarios/14d_set_C_recovery.py](scenarios/14d_set_C_recovery.py), [scenarios/14d_set_D_phase_shift.py](scenarios/14d_set_D_phase_shift.py), [scenarios/14d_set_E_overtrained.py](scenarios/14d_set_E_overtrained.py), [scenarios/14d_set_F_sedentary.py](scenarios/14d_set_F_sedentary.py), [bug_reports/2026-05-03_latex_vs_code_reconciliation.md](bug_reports/2026-05-03_latex_vs_code_reconciliation.md) (triage row added)
- **What changed:**
  - `scenarios/_common.py` is now a single ~150-line runner that:
    1. Looks up the scenario in `models.swat.simulation.scenario_presets()` (in-repo source of truth — the truth values match the canonical `PARAM_SET_*` / `INIT_STATE_*` from `~/Repos/SWAT_model_dev/`).
    2. Constructs a `StepwisePlant` with `DEFAULT_PARAMS` and the scenario's init state.
    3. Calls `plant.advance(stride_bins=14·BINS_PER_DAY, V_h_daily, V_n_daily, V_c_daily)` once → exercises the same SDE integrator + obs samplers the closed-loop MPC bench uses.
    4. Applies 5% Bernoulli dropout to HR + stress (sleep / steps preserved) via `psim.scenarios.missing_data.apply_dropout` (a small standalone numpy function — the only psim call left).
    5. Writes a packaged-style artefact to `outputs/swat/<scenario>/`: `trajectory.npz` + per-channel `obs/obs_*.npz`.
    6. Compares end-of-trial T to a **basin classifier** (qualitative: `high (>0.5)` / `collapsed (<0.2)` / etc.) and exits 0 if the basin matches, 1 if it doesn't. Returns "BASIN OK" or "BASIN MISMATCH" per scenario.
  - The 6 per-set scripts are now ~20 lines each — they import the runner and call it with one of the six string keys (`'A_healthy'`, `'B_amplitude_collapse'`, …).
  - The expected end-of-trial T values are deliberately **qualitative basins** rather than specific numbers. Reason: the LaTeX's old "T* ≈ 0.55 / ≈ 1.0" expected values were calibrated for an older `μ(E)` form pre the V_h-anabolic structural fix; the current healthy T* ≈ 0.84. Locking in numerical targets would either silently hide model evolution or fail spuriously on every parameter tune. A basin classifier (`high` vs `collapsed` vs `rising`) is invariant to those drifts and still loud if the dynamics truly invert.
- **Why:** Per the dev sandbox's purpose — validate the model files before shipping to `smc2fc` — `_plant.py` is the simulator-as-plant that drives the closed-loop MPC. If `_plant.py` or the obs samplers in `simulation.py` quietly drift, the closed-loop bench downstream will produce wrong results. The scenarios' job is to catch that. Previously they were broken because `scenarios/_common.py` reached into psim's `SDEModel`/`synthesise_scenario` machinery and asked for a `SWAT_MODEL` symbol that doesn't exist in this dev repo (it only existed in the canonical `~/Repos/SWAT_model_dev/`). The right fix is to use the dev repo's own leaner API (`StepwisePlant.advance()` already does forward-sim + obs sampling in one call), not to port the heavyweight `SDEModel` machinery in here.
- **Verified by:** Real fresh-install workflow run from `/tmp` (not from the dev repo dir):
  - All 6 scenarios run cleanly:
    - A_healthy:           T_end = 0.848 → BASIN OK (high, near T*)
    - B_amplitude_collapse: T_end = 0.066 → BASIN OK (collapsed)
    - C_recovery:          T_end = 0.781 → BASIN OK (rising past 0.5)
    - D_phase_shift:       T_end = 0.048 → BASIN OK (collapsed under jet-lag)
    - E_overtrained:       T_end = 0.492 → BASIN OK (reduced by V_n damping)
    - F_sedentary:         T_end = 0.132 → BASIN OK (collapsed by gate term)
  - Each writes a 1344-bin trajectory (14d × 96 bins/d at 15-min) + 4 obs streams (HR/sleep/steps/stress) to `outputs/swat/<scenario>/`.
  - 5% dropout reduces HR/stress sample counts by ~5% as expected (1344 → 1273 / 1266).
  - Existing 6 pytest tests remain green. No code in `models/swat/` was touched.

### 21:45 BST — Fix simulator: include `delta_HR` and `delta_s` in the synthesised observations
- **Author:** Ajay Talati
- **Assistant:** Claude Code (Opus 4.7, 1M context)
- **Files:** [models/swat/simulation.py](models/swat/simulation.py)
